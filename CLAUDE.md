## Domain design

Entities: `Doctor`, `Slot`, `Booking`, `Patient` (seeded, stands in for auth).

- **Slot**: `id, doctor_id, start_at, end_at`, unique `(doctor_id, start_at)`. Times stored UTC.
- **Availability is derived, not stored**: a slot is available iff it has no *active* booking.
  No `booked` boolean â€” a flag would be a second source of truth to keep consistent.
- **Booking**: `id (uuid), slot_id, patient_id, status, idempotency_key, version, created_at,
  updated_at`.
- Slots are generated by an idempotent startup seeder (next 14 days, business hours). A real
  system would have doctor-managed scheduling; out of scope.

### Booking state machine

| From      | Action   | To        |
|-----------|----------|-----------|
| â€”         | book     | PENDING   |
| PENDING   | confirm  | CONFIRMED |
| PENDING   | cancel   | CANCELLED |
| CONFIRMED | cancel   | CANCELLED |
| CONFIRMED | complete | COMPLETED |

CANCELLED and COMPLETED are terminal. Transitions are enforced **only** inside `Booking` domain
methods (`confirm()`, `cancel()`, `complete()`) that throw on invalid moves â€” no public status
setter, no transition logic in services or controllers.

### Concurrency defense (the centerpiece)

Layered, in order of authority:

1. **Partial unique index â€” the final arbiter.** Two concurrent inserts for the same slot cannot
   both commit, regardless of app instances or interleaving:

   ```sql
   CREATE UNIQUE INDEX ux_active_booking_per_slot
     ON booking (slot_id)
     WHERE status IN ('PENDING', 'CONFIRMED');
   ```

2. **Optimistic locking** (`@Version` on Booking) so concurrent state transitions
   (e.g. confirm vs cancel racing) can't clobber each other â€” loser gets 409.
3. **Graceful surfacing**: catch `DataIntegrityViolationException` / `OptimisticLockException`
   in one `@RestControllerAdvice`, return **409 Conflict** as RFC 7807 `problem+json`.

Alternatives to document (not implement) in README: pessimistic `SELECT FOR UPDATE` on the slot
row (simpler to reason about, serializes writers, hurts throughput); Redis/distributed locks
(extra infra, still needs the DB constraint as backstop anyway).

### Idempotency

`Idempotency-Key` header **required** on `POST /bookings` (400 if missing). Unique on
`(patient_id, idempotency_key)`; a replayed key returns the original booking instead of a
duplicate attempt. Cheap to build, strong real-world signal.

## API surface (v1)

Auth is stubbed: `X-Patient-Id` header selects a seeded patient. State it in every write-up.

| Method | Path                          | Notes                                             |
|--------|-------------------------------|---------------------------------------------------|
| GET    | /api/v1/doctors               | List doctors                                      |
| GET    | /api/v1/doctors/{id}/slots    | Available slots, `?from=&to=` (default: next 7d)  |
| POST   | /api/v1/bookings              | `{slotId}` + `Idempotency-Key` â†’ 201, or 409 taken |
| GET    | /api/v1/bookings              | Current patient's bookings                        |
| GET    | /api/v1/bookings/{id}         |                                                   |
| POST   | /api/v1/bookings/{id}/confirm | 409 on invalid transition                         |
| POST   | /api/v1/bookings/{id}/cancel  | 409 on invalid transition                         |

Errors: RFC 7807 `ProblemDetail` (built into Spring 6). 409 = slot taken or invalid transition;
422 = validation; 404 = unknown ids.

## Testing strategy

- **Unit**: state machine â€” every valid and invalid transition.
- **Integration (Testcontainers, real Postgres)**: booking happy path, idempotent replay,
  constraint mapping to 409.
- **The killer test (maps to the brief's bonus)**: `ExecutorService` + `CountDownLatch` fire
  ~50 simultaneous bookings (distinct patients/keys) at one slot. Assert **exactly one 201**,
  the rest 409, and exactly one active booking row in the DB. Second race test: concurrent
  confirm vs cancel â€” exactly one wins via optimistic lock.
- Run: `./mvnw test` from `backend/` (**Docker Desktop must be running** for Testcontainers â€”
  we develop on Windows; `mvnw.cmd` in PowerShell).

## Deployment â€” Render free tier (user already has an account)

Single **Web Service** from the root `Dockerfile` (frontend baked into the jar â†’ one URL, no
CORS, only one free service consumed) + a free **Render Postgres**.

Render-specific gotchas â€” respect these when writing config:

- **Free Postgres expires 30 days after creation** (then gets deleted). Create it shortly
  before submission, not at project start. Local dev never depends on it.
- Render hands out a `postgres://` URL; Spring needs JDBC. Set env vars explicitly on the
  service: `SPRING_DATASOURCE_URL=jdbc:postgresql://<internal-host>:5432/<db>`,
  `SPRING_DATASOURCE_USERNAME`, `SPRING_DATASOURCE_PASSWORD` (use the *internal* connection
  values from the Render dashboard).
- 512MB instance: set `JAVA_TOOL_OPTIONS=-XX:MaxRAMPercentage=75.0` and keep the runtime image
  a plain JRE (e.g. `eclipse-temurin:21-jre`).
- Health check path: `/actuator/health` (include `spring-boot-starter-actuator`).
- Free services **spin down after ~15 min idle; first request takes up to ~1 min** (JVM cold
  start). Put this note prominently in the README so reviewers don't think the app is broken.
- Flyway runs migrations on startup â€” no separate deploy step.

## Deliberate scope cuts (each gets a one-liner + "how I'd add it" in README)

## Build phases (each phase ends green and committable)

1. **Skeleton**: repo layout, compose, Spring Boot app boots, Flyway baseline migration.
2. **Domain**: entities + state machine with unit tests. DoD: all transition tests pass.
3. **Booking flow**: endpoints, idempotency, constraint + advice mapping. DoD: manual double-book
   attempt returns 409.
4. **Race tests**: Testcontainers concurrency tests. DoD: 50-thread test proves exactly one win.
5. **UI**: slot grid, book button, visible "slot already taken" error when losing the race.
6. **Deploy**: Dockerfile, Render service + Postgres, live URL end-to-end.
7. **Write-up**: README (setup, tests, API table, stack justification, assumptions/limitations,
   trade-offs incl. locking alternatives). This phase is graded â€” don't rush it.
